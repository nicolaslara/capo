capo refocus · control-plane decision brief

Making capo control native sub-agents:
the Bridge fork vs. the Spec change

Two ways to go beyond the fork-free Path-1 lockdown. One is recommended and feasible; the other is the option the protocol maintainers reject by design. This brief lays out both honestly so you can weigh them. Everything here traces to docs/capo-refocus/CONTROL-PLANE-RESEARCH.md (Phase-1, Phase-2 prototype results, Phase-1b).

Bottom line

  • DO NOW  Path-1 lockdown, no fork. Strip native tools/sub-agents via _meta.claudeCode.options and re-supply file/shell/search as capo MCP tools. Proven empirically (Proto-1 PASS) against the stock bridge. This is the product.
  • RECOMMENDED NEXT  Fork the bridge (claude-agent-acp) for observe / veto / steer / visualize over native sub-agents. Small, additive, _meta-shaped, plausibly upstreamable.
  • NOT WORTH IT  Forking the ACP spec. The only thing that would give per-sub-agent sessions — but maintainers reject sub-agent primitives by design. High cost, breaks interop. Watch session/fork instead.

Where the bridge sits in the stack

capo is an ACP client. It does not talk to claude directly — it spawns a bridge that translates the Agent Client Protocol into Claude Agent SDK calls, which drive the nested claude engine. That bridge is the piece in play.

claude-agent-acp is the npm package historically published as @zed-industries/claude-code-acp, now living upstream at agentclientprotocol/claude-agent-acp (renamed @agentclientprotocol/claude-agent-acp, current v0.33.x). Our references pin claude-code-acp@0.16.2 / @anthropic-ai/claude-agent-sdk@0.2.44; line numbers below are 0.16.2-specific and must be re-baselined to v0.33.x before any fork (Proto-6).

Option 1 The Bridge fork  recommended · feasible

Fork claude-agent-acp and add a small, additive, env-flag-gated patch set. This does not touch the closed SDK or the protocol spec — it only stops the bridge from dropping information the SDK already produces, and lets capo's responses carry richer semantics back. Five patches, each justified below.

Each patch is (why capo needs it · what it unlocks · upstreamability · cost/fragility). Markers: proven already observed empirically · upstreamable same shape as PRs maintainers already merge.

▸ 1  settingSources:[] override — close the ambient-settings leak  proven may need no patch

Why: The bridge hardcodes settingSources:["user","project","local"] (acp-agent.js:777), so a hostile or ambient .claude/settings.json / .mcp.json / CLAUDE.md — or the operator's own user-level MCP servers — can silently re-introduce tools and permissions, defeating the lockdown.

Unlocks: Lockdown integrity — the agent's tool set is exactly what capo declares, nothing leaks in.

Status: Already proven via _meta in Proto-1 — passing _meta.claudeCode.options.settingSources:[] does override the hardcoded list (the Phase-1 "unproven" claim is now PROVEN). The real leak it neutralized was user-level settings pulling in the operator's own MCP servers. So this likely needs no fork patch at all — it's a config key the bridge already spreads before its own override.

Cost/fragility: Near-zero; depends only on the spread-before-override ordering, which held in 0.16.2 and should be re-checked on v0.33.x.

▸ 2  Forward agentID into requestPermission._meta — attribute a sub-agent's inner call  proven drop upstreamable

Why: The SDK does supply the sub-agent's identity — sdk.mjs:21 passes agent_id in the can_use_tool control request, surfaced as CanUseTool.agentID (sdk.d.ts:120, "sub-agent's ID"). Proto-2 logged it populated: agentID:"ae69749" for a sub-agent's inner Bash. But the bridge drops it: acp-agent.js:~575 destructures only {signal,suggestions,toolUseID}, so it never reaches capo.

Unlocks: Attribution. Today capo can see and gate a sub-agent's inner call but cannot say which sub-agent made it. This one forward turns an anonymous nested call into "sub-agent A ran bash echo INNER."

Upstreamable: Yes — _meta.claudeCode.* attribution is exactly what the Completed meta-propagation RFD (2026-06-03) endorses, and "don't drop SDK events" PRs already land (#649/#676/#694/#713).

Cost/fragility: A one-line forward. Lands in the hot message-routing zone, so re-baseline to v0.33.x first.

▸ 3  Deny-message + interrupt:false passthrough — deny with guidance, not a turn-killing veto  blocker confirmed upstreamable

Why: The SDK supports deny-with-message — canUseTool can return {behavior:"deny", message, interrupt?} (sdk.d.ts:889–903); the engine writes message as an is_error:true tool_result the model reads next turn (Anthropic's verbatim "Suggest alternative" recipe). But the bridge's reject branch hardcodes message:"User refused permission to run tool" and interrupt:true (acp-agent.js:683–689). Two failures: capo's redirect never reaches the model, and interrupt:true aborts the whole turn — the opposite of steering. This is exactly the interrupt:true Proto-2 logged. The ACP wire has no free-text field on a permission response (SelectedPermissionOutcome carries only optionId); _meta is the only escape hatch, and the bridge ignores it. So this is a bridge patch, not config.

Unlocks: Steering. capo can deny a Task call and say "Task is disabled here — call mcp__capo__start_agent instead," so the model pivots at call time instead of failing silently or looping. This is the load-bearing mechanism behind the ban-Task-force-start_agent posture.

Patch shape:

// acp-agent.js reject branch — derive from capo's response instead of hardcoding
const denyMsg = response._meta?.["capo.denyMessage"]
                ?? "User refused permission to run tool";
return { behavior:"deny",
         message: denyMsg,
         interrupt: response._meta?.["capo.interrupt"] ?? false };

Upstreamable: Yes, _meta-shaped. Cost/fragility: a few lines, same hot file as patch 2 — folds into one fork effort. Confidence the message transports is HIGH; confidence the model reliably pivots is MEDIUM (prompt-following) — Proto-5 confirms.

▸ 4  Relax the subagent text/thinking streaming gate (parent_tool_use_id === null) — visualize a sub-agent's live steps  upstreamable

Why: The bridge streams text/thinking partials only when parent_tool_use_id === null (~lines 1288/1395 on v0.33.x), i.e. only for the top-level agent. A sub-agent's intermediate reasoning and steps are gated off and never reach capo as live partials. (Tool calls already arrive tagged with _meta.claudeCode.parentToolUseId on v0.33.x — this gate is specifically about live text/thinking.)

Unlocks: Visualization. capo can render a sub-agent's live steps in its sidebar — the Proto-4 FAIL mode — instead of seeing only the top-level Task call and its final result.

Upstreamable: Yes — additive and PR #713-shaped ("forward task events"). Cost/fragility: additive; tag relaxed chunks with parentToolUseId so capo can attribute them.

▸ 5  Register a PreToolUse hook callback — the universal observation point  fires for sub-agents needs fork, not config

Why: The PreToolUse SDK-callback hook is the most complete observation point — Proto-2 showed it fired for all three calls (Task, inner Read, inner Bash), including auto-allowed ones that bypass the permission round-trip. But SDK hooks are JS callbacks (sdk.d.ts:259–268); a callback cannot be serialized across JSON into _meta. So capo cannot inject this hook by config — the callback must be registered inside the bridge, which can then forward each event to capo over the wire. This is the hard reason the _meta-only "thin wrapper" idea is rejected by source.

Unlocks: Universal observation + backstop veto. capo sees every nested tool call — even auto-allowed ones the permission gate never sees — giving defense-in-depth telemetry over any native sub-agent that ever slips through.

Cost/fragility: The one patch that requires a fork (can't be config). Register the callback, forward (agent_id, tool_name, tool_use_id) to capo. Additive, env-flag-gated.

The durable ceiling. Even fully forked, sub-agents stay logical children inside ONE ACP session — they run in-process as sidechains with no isolation option, and ACP's SessionId is flat with no parent/child primitive. The bridge fork buys capo attribution, steering, visualization, and veto over native sub-agents — but it can never make them separate capo-controlled sessions. The only route to a real session per sub-agent is to ban native Task/Agent and force the conductor onto capo's start_agent (a Path-1 posture).
Why the fork stays cheap. Every patch is _meta-shaped and matches changes maintainers already accept (the meta-propagation RFD is Completed; "don't drop SDK events" PRs land routinely; responsive triager). Keep the diffs small, additive, and behind a CAPO_* env flag (repo precedent: CLAUDE_CODE_EMIT_SESSION_STATE_EVENTS), re-baseline to v0.33.x — and if upstreamed, fork-maintenance trends toward zero. The real cost is rebase churn: src/acp-agent.ts is one ~3,750-line hot file under frequent SDK bumps, and our diffs land in its hottest zone (the message-routing switch).

Option 2 The Spec change  maintainers reject by design

The bridge fork's ceiling is a property of the protocol: ACP has no way to model a sub-agent as a first-class session. The only way past that ceiling is to change the ACP spec itself (agent-client-protocol). Presented honestly so you can weigh it — but the maintainers have decided against it.

What the spec would need

  • A sub-session / parent-child session primitive — so a sub-agent could be its own SessionId nested under a parent, instead of a flat opaque id.
  • An agent_id / sub_session_id field on tool calls, session updates, and permission requests — first-class sub-agent attribution at the protocol level (not smuggled through _meta).
  • And/or the session/fork RFD — the one upstream proposal that could ever yield isolated sub-conversations.

Why capo would ideally want it

Per-sub-agent sessions are the dream end-state: capo could steer and visualize each sub-agent as its own first-class entity — a real session it owns, not a logical child it can only observe through _meta attribution. That is strictly more than the bridge fork can deliver.

Why the maintainers say no

  • Subagent semantics belong to the Claude Agent SDK, not ACP. The maintainer posture (issue #623) is explicit: ACP only relays; sub-agent dispatch and context inheritance live in the closed SDK (sub-agents don't even inherit CLAUDE.md).
  • No sub-agent primitive on any roadmap. Neither the v1 nor the v2 roadmap has a parent/child, sub-session, or agent_id field on tool calls/updates. This is a deliberate design boundary, not an oversight.
  • _meta is the blessed channel instead. The project actively endorses namespaced _meta for cross-protocol attribution (meta-propagation RFD Completed) — precisely so the core types don't grow vendor-specific sub-agent fields.

Trade-offs of forking the spec anyway

  • Multi-vendor governance. The spec moved to a vendor-neutral org (agentclientprotocol/agent-client-protocol) and moves fast; a unilateral core-type change has no path to consensus.
  • A divergent core type is unmergeable and perpetually stale. Forking a fundamental protocol type means rebasing the entire spec forever with no hope of upstream — the opposite of the bridge fork's additive _meta patches.
  • Breaks interop. The whole point of ACP is that any ACP client talks to any ACP agent. A capo-specific session model breaks that contract.
The one distant peg. The session/fork RFD (author @josevalim) explicitly names subagents as a use case — the single upstream primitive that could ever yield isolated sub-conversations. It is worth watching, but it is not shipping, and it is not a parent/child runtime tree. Don't build on it; track it.

Recommendation

Ship enforcement now, fork the bridge for the safety net, leave the spec alone.

  • 1 · Path-1 lockdown now — no fork.
    Strip native tools/sub-agents via _meta.claudeCode.options (disableBuiltInTools:true + settingSources:[] + disallowedTools:[Task,Agent,…]) and re-supply file/shell/search as capo MCP tools; the conductor delegates only through start_agent. Proven (Proto-1 PASS). This is the load-bearing layer and the actual product.
  • 2 · Bridge fork for observe / veto / steer.
    One additive, env-flag-gated fork of claude-agent-acp v0.33.x carrying: the agentID→_meta forward (attribution), the deny-message + interrupt:false passthrough (steering), the relaxed subagent-streaming gate (visualization), and the registered PreToolUse hook (universal observation). settingSources:[] is already proven and likely needs no patch. Re-baseline first (Proto-6); confirm the model actually pivots on a guided deny (Proto-5). This is the defense-in-depth backstop over any native sub-agent.
  • 3 · Spec change — not worth it.
    Forking the ACP spec is the only thing that yields per-sub-agent sessions, but maintainers reject sub-agent primitives by design and the cost (unmergeable core type, broken interop, perpetual staleness) far exceeds the value. Don't fork the spec — watch session/fork as the single distant peg.

Net: enforcement (Path-1) makes the common case impossible-to-violate by construction; the bridge fork makes the residual native-sub-agent case observable, attributable, steerable, and vetoable — the hybrid control plane the research set out to build. The spec change is the dream end-state on paper and a non-starter in practice.
